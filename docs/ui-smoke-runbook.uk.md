# UI smoke-runbook (10-15 хв)

Короткий сценарій ручної перевірки після деплою, щоб швидко підтвердити базову працездатність інтерфейсу.

## 1. Передумови

- Деплой завершено без помилок.
- Міграції застосовані (`No pending migrations`).
- Є щонайменше один користувач `ADMIN`.

## 2. Базова доступність

Відкрити в браузері:

- `/app/login`
- `/app/register`
- `/app/tournaments`
- `/app/leaderboard`

Очікування:

- сторінки відкриваються без runtime error;
- верхня панель містить `Назад`, `Головна`, `Дашборд`/`Реєстрація`.

## 3. Вхід і рольові дашборди

1. Увійти під `ADMIN`.
2. Перевірити `/app/admin`:
 - немає падіння сторінки;
 - блоки метрик рендеряться;
 - якщо даних немає, показується зрозумілий `empty-state`, а не «красиві нулі».
3. Перевірити `/app/profile`:
 - вкладки `Edit / Preferences / Security` відкриваються;
 - `Save` працює і показує системне повідомлення (toast + inline notice).

## 4. Ключові дії і нотифікації

Перевірити, що після успішних/помилкових дій з'являються toast-сповіщення:

- `ADMIN`: створення турніру, створення раунду, зміна статусу, розподіл призначень.
- `JURY`: збереження оцінки.
- `TEAM`: реєстрація команди, збереження сабміту.
- `PROFILE`: збереження налаштувань.

## 5. API-маршрути через проксі

Без токена:

- `GET /dashboard/admin/metrics` -> `401` JSON
- `GET /dashboard/jury/metrics` -> `401` JSON
- `GET /dashboard/team/metrics` -> `401` JSON
- `GET /profile/settings` -> `401` JSON

Це підтверджує, що Caddy маршрутизує запити в backend, а не повертає HTML фронтенду.

## 6. Що прикласти до звіту

- скрін `/app/admin` після логіну;
- скрін `/app/profile` після `Save`;
- короткий лог перевірки API (4 запити вище).
